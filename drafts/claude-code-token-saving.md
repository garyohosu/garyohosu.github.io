# 生成AIのトークン消費が増えすぎる理由と、Claude Codeでできる節約法

この記事では、Claude Codeを使っていると増えがちなトークン消費の理由と、今日からできる節約法をわかりやすく整理します。

---

## 1. 生成AIのトークン消費が増えている

最近、生成AIを使っていると、「便利だけど、思ったよりトークンを食う」と感じる場面が増えてきました。

特にClaude Codeのような**AIコーディングエージェント**は、普通のチャットAIとは事情が違います。
ただ質問に答えるだけでなく、会話履歴を保持し、ソースコードを読み、コマンドを実行し、テスト結果やエラーログを読み、複数ファイルを横断して調査し、MCPやsubagent(サブエージェント)といった仕組みを使い、長時間同じセッションで作業を続けます。

たとえるなら、AIコーディングエージェントは、1つの調べ物のために本を10冊も机に積んでしまう調査員のようなものです。
積んだ本は次の作業でもそのまま机に残り、片付けない限り視界を占領し続けます。
Claude Codeのトークン消費も、これと似た構造で増えていきます。

この記事では、なぜトークン消費が増えやすいのかを整理したうえで、Anthropic公式ドキュメントで紹介されている節約の考え方と、ユーザーの間でよく言われている経験則の両方を、初心者にもわかるようにまとめます。

---

## 2. そもそもトークンとは何か

トークンとは、AIが文章やコードを処理するときに使う単位です。
単語全体、単語の一部、記号、空白、コード中の断片などに分割されることがあります。
人間にとっての「文字数」や「単語数」と完全に一致するものではありません。

- **言語による違い**:日本語、英語、記号、プログラムのコードでは、同じ文字数でもトークンの数が違います。
- **入力トークンと出力トークン**:AIに送る内容が入力、AIが返してくる内容が出力で、どちらもトークンとして数えられ、料金や利用制限に関わります。
- **会話以外の入力**:コーディングエージェントでは、ユーザーが打った文章だけでなく、AIが読んだファイルの中身や実行したコマンドの結果もコンテキストに含まれます。利用可能なツールの名前や、実際に使用したツールの定義・実行結果なども、コンテキストに含まれる場合があります。ただしMCPツールの詳細な定義は[必要になるまで遅延読み込みされる仕組み](https://code.claude.com/docs/en/costs)があるため、「すべてが最初から毎回読み込まれる」と考えるのは正確ではありません(詳しくは8節で説明します)。
- **読み直しのコスト**:AIエージェントは、それまでの会話を踏まえて次の判断をするため、会話が長くなるほど毎回読み直す範囲も広がります。

つまり、一言だけ追加で聞いたつもりでも、裏側では過去の会話やファイルの情報を含めて、AIがもう一度まとめて読み直している場合があります。
これが、コーディングエージェントのトークン消費が体感より増えやすい理由のひとつです。

## 3. Claude Codeでトークン消費が増えやすい理由とよくある失敗

Claude Code特有の事情を、公式ドキュメント([Best practices for Claude Code](https://code.claude.com/docs/en/best-practices)、[Manage costs effectively](https://code.claude.com/docs/en/costs))の内容をもとに整理します。

- **長いセッションの継続**:会話は、それまでのメッセージ、読んだファイル、実行結果をすべて含んだ**コンテキストウィンドウ**の中で処理されます。公式ドキュメントでも、1回のデバッグセッションやコードベースの探索だけで数万トークンを消費することがある、と説明されています。
- **大きなファイルの読み込み**:ファイルの内容はそのままコンテキストに乗るため、大きなファイルほど負荷が大きくなります。
- **長大なログ**:テストログ、ビルドログ、エラーログは冗長になりがちで、必要な情報はごく一部ということも多いです。
- **曖昧な依頼**:スコープを絞らずに「調査して」と頼むと、Claudeが何百ものファイルを読んでコンテキストを埋めてしまう、という失敗パターンが公式ドキュメントに明記されています。
- **読み込ませる情報の肥大化**:MCPサーバー、CLAUDE.md、hooksやskillsの設定によって、毎回のコンテキストが重くなることがあります(詳しくは8節で説明します)。
- **拡張思考(extended thinking)や高性能モデル**:複雑な計画や推論のタスクで性能を大きく高めますが、単純作業では過剰になることがあります。思考に使ったトークンも出力トークンとして課金対象になるためです。
- **subagentやagent team**:複数のエージェントで協調作業をする仕組みで、強力な分、それぞれが独自の文脈を持つためトークン消費が増えやすくなります(詳しくは10節で説明します)。

これらの原因は、そのまま日常的な失敗パターンにもつながります。

- ずっと同じセッションで何日も作業を続ける(不要な文脈が積み上がる)
- 何でも巨大ログを丸ごと貼る
- 「いい感じにして」とだけ依頼する(広範囲の探索を招きやすい)
- MCPサーバーやCLAUDE.mdの中身を整理せず放置する
- 毎回、最高性能のモデルと最高レベルのthinkingで作業する
- `/compact`や`/clear`の前に、重要な決定事項をファイルに残していない
- トークン節約を意識しすぎて、AIの判断に必要な情報まで削ってしまう

最後の項目は見落とされやすいものです。
節約は「なるべく何も渡さない」ことではありません。
AIの判断に必要な情報まで削ると判断の精度が落ち、修正の往復が増えて、かえって多くのトークンを使うことになります。

## 4. まず使用量を見える化する

節約の第一歩は、気合いではなく計測です。
何にどれだけ使っているかわからないままでは、どこを削ればいいのかも判断できません。

Claude Codeでは、[`/usage` コマンド](https://code.claude.com/docs/en/costs)で使用状況を確認できます。

- 画面上部の「Session」欄に、現在のセッションのトークン使用量や推定コストが表示されます。ただしこの金額は、ローカルのトークン数から計算した目安であり、実際の請求額とは異なる場合があります。正確な金額は、Claude ConsoleのUsageページで確認する必要があります。
- Pro、Max、Team、Enterpriseなどのプランでは、プランの上限に対する使用状況の内訳(skills、subagents、plugins、個々のMCPサーバーごとの使用割合など)も表示されます。`d`キーと`w`キーで、直近24時間と直近7日間の表示を切り替えられます。
- API利用とサブスクリプション(Pro/Max/Team/Enterprise)利用とでは、表示される情報の意味が異なります。サブスクリプションでは、セッションのコスト表示自体が課金に直結するものではなく、プランの利用バーの方が実際の目安になります。

まずは `/usage` を開いて、自分がどこにトークンを使っているのかをざっと眺めてみることをおすすめします。

## 5. コンテキストを整理する:`/clear`と`/compact`

Claude Codeには、会話の文脈を整理するための代表的な2つのコマンドがあります。
[`/clear`と`/compact`の使い分け](https://code.claude.com/docs/en/best-practices)は、公式ドキュメントでも重要な運用ポイントとして紹介されています。

`/clear` は、会話履歴を丸ごとリセットするコマンドです。
イメージとしては、机の上を一度全部片付ける操作です。

使うとよい場面は、次のようなときです。

- まったく別の作業に移るとき
- 前の調査内容がもう不要になったとき
- 試行錯誤で会話が汚れてきたと感じるとき
- Claudeが古い前提に引っ張られていると感じるとき

公式ドキュメントでは、同じ問題を2回以上訂正しているなら、そこで`/clear`して、学んだことを踏まえたより具体的なプロンプトで仕切り直したほうがよい、という指針が示されています。
訂正を重ねるほど、会話の中に「うまくいかなかったやり方」の情報が積み上がり、それがかえってAIの判断を鈍らせるためです。

注意点もあります。

- `/clear`は会話を完全に消すため、進行中の重要な前提や決定事項まで消えてしまいます。必要な情報は、事前にメモやREADME、TODOファイルなどに残しておくと安心です(なおCLAUDE.mdはセッションごとに自動で再読み込みされるため、`/clear`後も引き続き読み込まれます)。
- 何でもかんでも`/clear`すると、逆に同じ説明を何度もAIにし直すことになり、非効率になる場合もあります。
- `/rename`で今のセッションに名前を付けてから`/clear`し、後で`claude --resume`(または`/resume`)で呼び戻す、という使い方も公式ドキュメントで紹介されています。

`/compact` は、長くなった会話を要約して、重要な情報だけを残すためのコマンドです。

`/clear`が「全部片付ける」だとすると、`/compact`は「要点だけノートにまとめる」というイメージです。
長時間の作業、デバッグ、リファクタリングのように、同じ作業を続けながら文脈を整理したい場面で役立ちます。

指示付きで使うこともできます。

```
/compact 今回の変更点、未解決のエラー、次にやる作業を中心に残してください
```

Claude Codeはコンテキストの上限が近づくと、自動でも要約(auto-compaction)を行います。
それでも、`/compact`を自分のタイミングで実行すれば、何を優先して残すかをある程度コントロールできます。

ただし、要約である以上、細かい情報が落ちる可能性はあります。
重要な仕様や決定事項は、`/compact`任せにせず、ファイル(README、TODO、設計メモなど)に残しておくのが安全です。

どちらのコマンドを使う前にも、[`/context`](https://code.claude.com/docs/en/context-window)でいま何がコンテキストを占めているかを確認しておくと、削るべき対象を判断しやすくなります。

使い分けの目安は、次のとおりです。

- 別の作業に移るなら `/clear`
- 同じ作業を続けるなら `/compact`
- 迷ったら `/context` でまず現状を確認する
- 重要な仕様や決定事項はファイルに残す

## 6. プロンプトを具体的にする

曖昧な依頼は、Claudeが「とりあえず広く調べる」という動きにつながりやすく、結果としてトークン消費も増えます。

悪い例:

```
このプロジェクトをいい感じに直して
```

良い例:

```
src/auth.ts の login 関数に入力チェックを追加してください。
メールアドレスが空の場合、パスワードが8文字未満の場合はエラーを返してください。
既存のテスト構成に合わせてテストも追加してください。
```

公式ドキュメントでも、ファイル名や状況、期待する挙動を具体的に書いたプロンプトの方が、修正の往復が少なく済むと紹介されています。
「AIに丸投げする」のではなく、「作業指示書を渡す」イメージを持つとうまくいきやすくなります。

ファイル名、関数名、目的、期待する結果、制約を書くほど、Claudeが探索する範囲は狭くなります。
結果として、トークン消費が減るだけでなく、精度も上がりやすくなります。

なお、曖昧なプロンプトが常に悪いわけではありません。
「このファイルで改善できそうなところは?」のような投げかけは、自分では気づかなかった観点を見つけたいときには有効です。

## 7. 巨大なログをそのまま貼らない

テストログ、ビルドログ、`npm install`のログ、CIのログなどは、長くなりやすい情報です。
全部貼ると、そのほとんどが不要な情報になってしまいます。

意識したいポイントは、次のとおりです。

- エラーの前後だけを渡す
- `grep`、`head`、`tail`などのコマンドで絞り込む
- Claude Codeに、必要な部分だけをコマンドで抽出させる

例:

```bash
npm test 2>&1 | grep -A 20 -B 5 "ERROR"
```

```bash
pytest 2>&1 | tail -100
```

「全部見せる」のは一見親切に見えますが、AIにとっては重い弁当箱をそのまま渡されるようなものです。
必要なところだけを渡すほうが、安くて速く、しかも正確です。

公式ドキュメントでも、hooks(8節で説明します)を使って「テストログからエラー行だけをgrepして返す」といった前処理の仕組みが紹介されており、この考え方はClaude Code自体にも組み込まれています。

## 8. 読み込ませる情報を減らす:MCP、CLAUDE.md、hooksとskills

Claude Codeに読み込ませる情報そのものを絞り込むことも、有効な節約策です。
ここではMCPサーバー、CLAUDE.md、hooksとskillsの3つを扱います。

**MCPサーバー**は、外部サービスとClaude Codeをつなぐ仕組みです。
便利な反面、使わないサーバーまで大量に有効化すると、管理やコンテキストが重くなりやすくなります。
[公式ドキュメント](https://code.claude.com/docs/en/costs)によると、MCPツールの詳細な定義は既定で「実際にそのツールを使う場面になるまで読み込まれない」形になっています。
とはいえ、有効なMCPサーバーが増えるほどツール名の一覧など基本的な情報は増えていきますし、実際に使う場面が増えれば、そのぶん文脈も増えます。
確認や整理には、`/mcp`で現在設定されているサーバーを確認し、`/context`で何がコンテキストを消費しているかを見た上で、使っていないサーバーを無効化するとよいでしょう。
GitHub操作など一部の外部サービス連携は、`gh`のようなCLIツールの方が、MCPよりも文脈効率がよい場合があります(CLIはツール一覧としての情報を追加しないため)。

**CLAUDE.md**は、プロジェクトの重要な指示をAIに伝えるための特別なファイルで、セッションのたびに自動で読み込まれます。
便利な反面、何でも書きすぎると、どんな作業でも毎回の基本コンテキストが重くなってしまいます。
[公式ドキュメント](https://code.claude.com/docs/en/costs)では、CLAUDE.mdは200行程度に収め、本当に必要なことだけを書くことが目安として紹介されています。
特定の作業でしか使わない長い手順は、CLAUDE.mdではなく後述のskillsに分けたほうが、基本コンテキストを軽く保てます。
各行について「これを消したらClaudeが間違えるか」と自問し、答えがNoなら削る、という考え方も紹介されています。

**hooks**と**skills**は、どちらもClaude Codeの拡張機能ですが、役割が異なります。

- **hooks**:特定のタイミングで自動的にスクリプトを実行する仕組みです。テスト結果をClaudeに渡す前に、失敗した部分だけをgrepで抜き出す、といった前処理に使えます。[公式ドキュメント](https://code.claude.com/docs/en/costs)には、テストコマンドの出力を失敗行だけに絞り込むPreToolUseフックの例が載っており、1万行のログを丸ごと読ませる代わりにフックがERRORにマッチする行だけを返すことで、コンテキストを数万トークンから数百トークンまで減らせる、と説明されています。
- **skills**:特定の作業のときだけ読み込まれる、専門的な手順や知識のまとまりです。CLAUDE.mdのように毎回読み込まれるのではなく、必要になったときだけ読み込まれるため、普段の基本コンテキストを軽く保てます。

MCPは工具箱のようなものです。
便利だからといって工具を全部机に出しっぱなしにすると、かえって作業がしづらくなります。
必要な工具だけを出しておくのが、この節に共通する考え方です。

## 9. モデルとthinkingを使い分ける

Claude Codeでは、モデルの種類と**extended thinking**(拡張思考)の深さを、状況に応じて調整できます。

難しい設計判断、複雑なバグ解析、アーキテクチャの検討では、高性能なモデルや深い思考が役立ちます。
一方で、単純な整形、軽微な修正、テストの追加、ログの抽出などでは、そこまでの性能は過剰になる場合があります。

[公式ドキュメント](https://code.claude.com/docs/en/costs)では、次のような調整方法が紹介されています。

- `/model`:使用するモデルを切り替える。Sonnetは多くのコーディング作業をこなせて、Opusより費用を抑えやすいとされ、Opusは複雑なアーキテクチャ判断や多段階の推論が必要な場面向けとされています。
- `/effort`:推論(思考)の深さの目安を切り替える。[公式のEffortページ](https://platform.claude.com/docs/en/build-with-claude/effort)では、低い方から順に、簡単で答えが明確なタスク向け、日常的な開発作業向け、複雑なデバッグや複数ファイルにまたがる修正向け、というように使い分けの目安が示されています。ごく難しい問題向けの、さらに深い設定が用意されている場合もあります。
- `/config`でthinking(拡張思考)自体をオフにする、あるいは環境変数で思考に使うトークンの上限を下げる、といった調整も可能です(モデルによって対応状況が異なります)。

何でも最高設定にするのは、ハンマーで豆腐を切るようなものです。
タスクの難しさに応じて設定を選ぶことが、節約と品質のバランスを取るコツになります。

注意点として、利用できるモデルや`/effort`の選択肢はプランや時期によって変わります。
実際に試す前に、必ず現在の公式ドキュメントを確認してください。
また、節約しすぎて品質が落ち、やり直しが増えると、かえって高くつくこともあります。

## 10. subagentとagent teamは使いどころを選ぶ

**subagent**(サブエージェント)は、メインの会話とは別の、独立したコンテキストで動く助手のような仕組みです。
調査やログ解析、別の観点からのレビューなどに向いています。

[公式ドキュメント](https://code.claude.com/docs/en/best-practices)では、Claudeがコードベースを調べるとき、大量のファイルを読むとそのすべてが自分(メインの会話)のコンテキストを消費してしまう、と説明されています。
subagentは別のコンテキストウィンドウで動き、要約だけを報告してくれるため、この消費を避けられます。

有効な使い方の例:

```
サブエージェントを使って、認証まわりのトークンリフレッシュの仕組みと、
既存のOAuthユーティリティが使えるかどうかを調査してください
```

大量のログ分析や広いコード調査、ドキュメント調査では、subagentが有効です。
一方で、1ファイルの軽微な修正のような小さな作業に毎回subagentを立てるのは、かえって過剰になることがあります。
subagentにも独自のコンテキストがあり、それを起動して維持すること自体がトークンを消費するためです。

複数のsubagentやセッションを同時に協調させる**[agent team](https://code.claude.com/docs/en/agent-teams)**は、特に強力な分、コンテキストの重複が大きくなります。
[公式ドキュメント](https://code.claude.com/docs/en/costs)では、planモードで動くteammateがいる場合、通常セッションの約7倍のトークンを使うことがあると説明されています。
そのため、チームは小さく保ち、役目が終わったteammateは早めに終了させることが推奨されています。

## 11. 作業単位を小さくする

「全部作って」と一度に丸ごと依頼するより、「まず認証部分だけ」「次にテスト」「最後にREADME」のように作業を分けて依頼したほうが、方向性のズレを早く見つけられます。

ただし、細かすぎる指示を連続で出すのも、それはそれで会話が長くなり、コンテキストが積み上がっていきます。
1回の依頼には、必要な情報をまとめて渡すことが大切です。

大きなタスクに取りかかるときは、[公式ドキュメント](https://code.claude.com/docs/en/best-practices)でも、まずplanモードで計画を立てさせ方向性を確認してから実装に入るという4段階の進め方(調査、計画、実装、コミット)が推奨されています。
方向性が間違ったまま実装まで進んでしまうと、大量の修正が無駄になりやすいためです。

早めに気づき、早めに軌道修正することも大切です。
Claude Codeには、作業を途中で止める`Esc`キーや、会話とコードを過去の状態に戻す`/rewind`(またはEscの2回押し)といった機能があります。
これらを使えば、うまくいかない方向に進んだときに、そこまでの文脈を丸ごと捨てずに仕切り直せます。

なお、使える機能や挙動はバージョンによって変わるため、実際に使う前に公式ドキュメントを確認することをおすすめします。

---

## 12. 実践用チェックリストとまとめ

そのままコピーして使えるチェックリストです。

- [ ] 作業前に、目的と対象ファイルを明確にしたか
- [ ] 不要な古い文脈が残っていないか
- [ ] 別作業に移るなら `/clear` したか
- [ ] 長時間の作業なら `/compact` したか
- [ ] `/usage` で使用量を確認したか
- [ ] `/context` で何が重くなっているか確認したか
- [ ] 使っていないMCPを無効化したか
- [ ] CLAUDE.mdが長すぎないか
- [ ] ログは必要な部分だけ渡しているか
- [ ] 単純作業に高コストな設定(高いモデルや高いeffort)を使っていないか
- [ ] 重要な仕様や決定事項をファイルに残したか

トークン節約は、AIをケチって使うことではありません。
AIに必要な情報だけを渡し、不要な情報を減らす整理術です。

Claude Codeは強力なツールですが、長時間の作業では文脈がどうしても太りやすい性質を持っています。
`/usage`で計測し、`/clear`と`/compact`で整理し、`/context`で何が重いのかを確認する。
この4つを使いこなすだけでも、感覚はかなり変わってくるはずです。

トークン節約の本質は、一言で言えば「AIの作業机を散らかさないこと」です。

※Claude Codeの仕様、料金、利用制限は変更される可能性があります。実際に使う前に、必ずAnthropic公式ドキュメントを確認してください。
